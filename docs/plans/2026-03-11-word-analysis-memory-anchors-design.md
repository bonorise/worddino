# WordDino 单词分析记忆锚点设计

## 1. 目标
- 在当前单词结果页中增加一层更直接的“记忆锚点”信息，帮助用户更快形成可视化记忆。
- 复用 `ljg-explain-words` 中“原始画面 + 核心意象 + 收束句”的分析方法，但不把产品带偏到“语言哲学解词”方向。
- 保持当前 `WordDino` 的核心定位不变，即“词根拆解 + 助记法 + 同族词关系图”的学习闭环。

## 2. 当前上下文
- 当前结果页已展示 `explanation`、`morphemes`、`mnemonics`、`examples`、`familyWords`。
- 当前 Gemini 返回结构对应 `src/lib/ai/gemini-schema.ts`，前端类型对应 `src/types/index.ts`。
- 当前 prompt 要求输出简单解释、可选拆词、助记卡片和例句，但缺少更高层的“记忆压缩层”。
- 当前产品定位明确偏向初学者和结构化记忆，不适合引入大段哲学化文案。

## 3. 方案对比

### 3.1 方案 A：最小增量引入记忆锚点
- 新增 `scene`、`formula`、`hook` 三个字段。
- 在结果页增加一张“记忆锚点”卡片，显示实际返回的字段。
- 优点是改动小、风险低、与现有页面结构兼容。
- 缺点是标题区信息密度提升有限，不解决音标和短释义问题。

### 3.2 方案 B：完整引入扩展分析头部
- 新增 `scene`、`formula`、`hook`、`gloss`、`ipa`。
- 结果页标题区同步增强。
- 优点是信息更完整。
- 缺点是需要额外处理音标可靠性和标题区布局，改动面变大。

### 3.3 方案 C：只改 prompt，不改 schema
- 让模型把“画面感”和“公式感”塞进现有 `explanation` 或 `mnemonics`。
- 优点是开发量最小。
- 缺点是信息结构混杂，无法稳定展示，也不利于后续迭代。

## 4. 核心决策
- 采用方案 A。
- 本轮只新增 `scene`、`formula`、`hook` 三个字段。
- `gloss` 和 `ipa` 暂不引入，避免把一次轻量改造扩展成标题区重构。
- 继续保留现有 `explanation`、`morphemes`、`mnemonics`、`examples`、`familyWords` 结构。

## 5. 字段设计

### 5.1 新增字段
- `scene?: string`
  - 含义：这个词最容易想象的原始画面或动作画面。
  - 约束：一句话、具体、可视化。
- `formula?: string`
  - 含义：把单词意义压缩成可记忆的短公式。
  - 约束：一句话或一行，不展开成长段解释。
- `hook?: string`
  - 含义：帮助用户收束记忆的一句话。
  - 约束：简短、实用、不鸡汤。

### 5.2 保持不变的字段
- `explanation`
- `decomposable`
- `morphemes`
- `mnemonics`
- `recommendedType`
- `examples`
- `familyWords`
- `source`

## 6. Prompt 设计
- 模型继续使用当前 locale 输出自然语言字段。
- 新增生成要求：
  - `scene`：提供一句能立即形成脑内画面的描述。
  - `formula`：提供一句短公式，帮助用户把意义压缩成结构。
  - `hook`：提供一句记忆收束句，强调“怎么记住它”。
- 风格约束：
  - beginner-friendly
  - concrete, visual, practical
  - 优先解释“结构如何通向现代义”
  - 禁止哲学化、诗化、鸡汤化表达
  - 禁止编造词源
- 容错规则：
  - 即使 `decomposable=false`，仍允许返回 `scene`、`formula`、`hook`
  - 如果模型无法稳定生成某字段，可省略该字段，不强制填空

## 7. UI 设计

### 7.1 新卡片位置
- 放在结果页主解释卡片之后。
- 再往下仍保留现有两栏布局和已有卡片。

### 7.2 卡片内容
- 标题：`记忆锚点`
- 子项顺序：
  - `原始画面`
  - `记忆公式`
  - `一句记住`

### 7.3 渲染规则
- 只要 `scene`、`formula`、`hook` 任一存在，就渲染整张卡片。
- 某个字段缺失时，不显示该子块。
- 三个字段都缺失时，不显示空卡片。

## 8. 测试设计

### 8.1 单元测试
- 更新 schema 测试，验证新字段可选且类型正确。
- 更新 analyzer 测试，验证 Gemini 返回新字段后可正确映射。

### 8.2 端到端测试
- 更新搜索流 e2e，用 mock `/api/analyze` 返回 `scene`、`formula`、`hook`。
- 断言结果页可以看到“记忆锚点”卡片及其内容。

## 9. 错误处理与降级
- 如果模型不返回新字段，结果页不应报错。
- 如果只返回部分新字段，页面只显示返回的部分，不显示空壳。
- 如果 schema 或接口解析失败，沿用当前结果页错误态。

## 10. 范围边界

### 10.1 本轮包含
- `scene`、`formula`、`hook` 字段设计与接入。
- Gemini prompt 调整。
- 结果页新增“记忆锚点”卡片。
- 中英文 i18n 文案补充。
- 单测与 e2e 更新。

### 10.2 本轮不包含
- `gloss`
- `ipa`
- 结果页标题区改造
- 新的助记类型
- 数据库存储与长期缓存策略

## 11. 验收标准
- 当 `/api/analyze` 返回 `scene`、`formula`、`hook` 时，结果页正确展示“记忆锚点”卡片。
- 当这三个字段全部缺失时，页面正常渲染且不显示空卡片。
- 现有解释、词根拆解、助记卡片、例句、同族词关系图仍保持可用。
- 单元测试和 e2e 测试通过。
